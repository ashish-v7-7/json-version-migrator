JSON Version Migrator

Chain-based migration framework for evolving versioned JSON stored in database columns. Auto-migrates any old version to the latest schema on read. Works as a Spring Boot starter or standalone.

⛓

Chain-Based

Each migration handles one step (v1→v2). The framework chains them to jump multiple versions.

⚙

Auto-Discovery

Drop a @Component class. Spring picks it up. No registration needed.

🛡

Fail-Fast Validation

Missing steps in the chain = app won't start. No silent runtime failures.

📚

Read-Only Migration

DB rows stay untouched. Migration happens in memory on read. Zero downtime.

Quick Start

1

Add the dependency

Add the Maven/Gradle dependency shown above.

2

Create a migration class

java@Component
public class CredentialV1ToV2 extends AbstractJsonVersionMigration {

    @Override
    public String jsonType()  { return "CREDENTIAL"; }

    @Override
    public int fromVersion() { return 1; }

    @Override
    public int toVersion()   { return 2; }

    @Override
    protected void applyChanges(DocumentContext doc) {
        doc.put("$.fields", "gateway_mode", "live");
    }
}
3

Inject and use

java@Service
public class MyService {

    private final JsonMigrationService migrationService;

    public Config readConfig(String rawJson) {
        String migrated = migrationService.migrateToLatest("CREDENTIAL", rawJson);
        return objectMapper.readValue(migrated, Config.class);
    }
}
That's it! When v1 JSON is read, the framework applies v1→v2 automatically. Add more steps and it chains them: v1→v2→v3→...→latest.

How It Works

DB Row
v1 JSON
→
migrateToLatest()
→
v1→v2
→
v2→v3
→
v3 JSON
→
Deserialize
  1. Service calls migrateToLatest("CREDENTIAL", rawJson)
  2. Framework reads $.version from the JSON → e.g., 1
  3. Registry looks up the latest version for "CREDENTIAL" → e.g., 3
  4. Registry builds the chain: [V1→V2, V2→V3]
  5. Each step mutates the DocumentContext in sequence
  6. Returns the migrated JSON string (with "version": "3")
Read-only The database row is never modified. Migration happens in memory. If you want to persist the migrated version, write it back explicitly.

JsonVersionMigration (interface)

javapublic interface JsonVersionMigration {
    String jsonType();       // your type identifier string
    int    fromVersion();    // source version
    int    toVersion();      // must be fromVersion() + 1
    void   migrate(DocumentContext doc);
}

AbstractJsonVersionMigration (base class)

Recommended. Auto-updates $.version after your changes. You only implement applyChanges(doc).

javapublic abstract class AbstractJsonVersionMigration
        implements JsonVersionMigration {

    public final void migrate(DocumentContext doc) {
        applyChanges(doc);
        doc.put("$", "version", String.valueOf(toVersion()));
    }

    protected abstract void applyChanges(DocumentContext doc);
}

JsonMigrationRegistry

Auto-discovers migrations, validates chains, provides lookup.

MethodDescription
init()Index + validate. Called automatically by Spring, or manually.
getLatestVersion(type)Highest version for a type (default: 1).
getRegisteredTypes()All types that have migrations.
getMigrationChain(type, from)Ordered list of steps from from to latest.

JsonMigrationService

The public API. This is what you inject.

MethodDescription
migrateToLatest(type, json)Migrate to the latest version. Returns unchanged if already latest.
migrateToVersion(type, json, target)Migrate to a specific version (not necessarily latest).
needsMigration(type, json)Check if migration is needed without doing it.
getCurrentVersion(json)Read the version number from a JSON string.

Recipe: Add a Field

SAFE Non-breaking. Old consumers ignore unknown fields.

javaprotected void applyChanges(DocumentContext doc) {
    doc.put("$.fields", "gateway_mode", "live");
}

Recipe: Remove a Field

BREAKING Ensure no code reads this field first.

javaprotected void applyChanges(DocumentContext doc) {
    try { doc.delete("$.fields.legacy_token"); }
    catch (Exception ignored) {}
}

Recipe: Rename a Field

BREAKING Copy to new key, then delete old.

javaprotected void applyChanges(DocumentContext doc) {
    Object val = doc.read("$.fields.key_id");
    if (val != null) {
        doc.put("$.fields", "api_key_id", val);
        doc.delete("$.fields.key_id");
    }
}

Recipe: Restructure / Nest

ADVANCED Move flat fields into a nested object.

javaprotected void applyChanges(DocumentContext doc) {
    Object clientId = doc.read("$.fields.client_id");
    Object secret   = doc.read("$.fields.client_secret");

    Map<String, Object> oauth = new LinkedHashMap<>();
    if (clientId != null) oauth.put("client_id", clientId);
    if (secret != null)   oauth.put("client_secret", secret);

    doc.put("$.fields", "oauth", oauth);
    doc.delete("$.fields.client_id");
    doc.delete("$.fields.client_secret");
}

Spring Boot Usage

Just add the dependency. The auto-configuration creates JsonMigrationRegistry and JsonMigrationService beans. All JsonVersionMigration @Component beans are auto-discovered.

No @EnableXxx annotation needed.

Non-Spring Usage

javavar registry = new JsonMigrationRegistry(List.of(
    new CredentialV1ToV2(),
    new CredentialV2ToV3()
));
registry.init();

var service = new JsonMigrationService(registry);
String migrated = service.migrateToLatest("CREDENTIAL", rawJson);

Custom Version Path

By default, the version is read from $.version. Override it:

java// Read version from $.schema_version instead
var service = new JsonMigrationService(registry, "$.schema_version");
Important When using a custom path, your migration classes must write to the same path in their migrate() method. If you extend AbstractJsonVersionMigration, it always writes to $.version. Implement the interface directly for custom paths.

Rules & Constraints

Sequential Only

toVersion() must equal fromVersion() + 1. No skipping.

No Gaps

Every version in the chain must be covered. Gaps = startup failure.

One Step Per Class

Each class handles exactly one transition.

Must Be Discoverable

Spring: annotate with @Component. Non-Spring: pass to registry constructor.

Types Are Independent

"CREDENTIAL" and "BRANDING" have separate chains. No interference.

Null-Safe

Null/blank JSON returns as-is. No exceptions.

Testing Migrations

java@Test
void testV1ToV2() {
    CredentialV1ToV2 migration = new CredentialV1ToV2();

    String v1 = """
        {"version":"1.0","fields":{"key_id":"rzp_123"}}
        """;

    DocumentContext doc = JsonPath.parse(v1);
    migration.migrate(doc);

    assertThat(doc.read("$.version", String.class)).isEqualTo("2");
    assertThat(doc.read("$.fields.gateway_mode", String.class)).isEqualTo("live");
}

FAQ

What if the chain has a gap?

The app fails to start with IllegalStateException. This is intentional.

Does it modify the database?

No. Migration is read-only (in memory). Write back explicitly if needed.

Can I inject Spring beans into migrations?

Yes. Migrations are @Component beans. But keep them simple — prefer pure JSON transforms.

What about performance?

Near-zero for up-to-date JSON (short-circuit on version check). A few microseconds per step for old JSON.

Can two types share a migration?

No. Each migration belongs to one type. Write separate classes if two types need the same change.

What Java/Spring versions are required?

Java 17+. Spring Boot 3.x (optional). Also works without Spring.
