1 Executive Summary

A Spring Boot starter library that auto-migrates versioned JSON stored in database columns from any old version to the latest schema on read, validates JSON structure on reads and writes, validates version correctness before saving, and optionally returns typed POJOs instead of raw JSON. Eliminates batch migration scripts, downtime, and if/else version handling in application code.

6 Key Decision Points for Team Review

Decision 1: Migration is read-only (lazy, on-demand) JSON is migrated in memory when read. The database row is never modified. This means zero downtime, zero batch scripts. Trade-off: first read of an old row is slightly slower (microseconds per step). If you need to persist migrated JSON, write it back explicitly.
Decision 2: Fail-fast at startup for broken chains If a migration step is missing (e.g., V2→V3 exists but V3→V4 does not, yet V4→V5 does), the application refuses to start. This prevents silent data corruption at runtime. Every developer must maintain a complete chain.
Decision 3: Validation is opt-in Schema definitions are optional. If you don't define one for a JSON type, validation always passes. This allows incremental adoption — start with migration only, add schemas when ready.
Decision 4: String-based type identifiers (not enum) JSON types are plain strings (e.g., "CREDENTIAL") instead of an enum. This means the library doesn't need to know your types — each team defines their own. Recommended: use a constants class (JsonTypes.CREDENTIAL).
Decision 5: Two write-path modes — strict latest vs any version with schema validateInput() requires the latest version (strict). validateInputAnyVersion() accepts any version that has a registered schema (flexible). Use strict for APIs that should only accept latest, flexible for services that accept multiple versions during gradual rollouts.
Decision 6: Typed POJO return is optional (via JsonDeserializer<T>) All migration methods return String by default. Pass a JsonDeserializer<T> to get a typed POJO back. The library has zero Jackson dependency — you provide the deserialization function. A JacksonDeserializer helper is included for convenience.
Decision 7: Per-version schemas enable multi-version validation You can register schemas for every version (V1Schema, V2Schema, V3Schema, etc.) — not just the latest. validateInputAnyVersion() reads the declared version from JSON, finds the matching schema, and validates against it. This enables validating old-format data without forcing migration first.

7 Benefits

Zero Downtime Migrations No batch scripts, no scheduled maintenance windows. Deploy the code and old data auto-migrates on read.
Structural Safety Schema validation catches corrupt data, missing fields, and type mismatches before they reach your DTOs.
Developer Simplicity Adding a new version = 1 migration class (~10 lines) + 1 schema class (optional). Framework handles chaining, version parsing, validation.
Audit Trail Each migration class is a self-documenting record of what changed and why. Git blame shows who changed what.
Typed POJO Return One call: migrate + validate + deserialize → typed DTO. No manual objectMapper.readValue() step. TypedMigrationResult<T> gives you both the POJO and validation result.
Multi-Version Input Validation Define schemas for all versions. validateInputAnyVersion() validates JSON at whatever version it declares. Catches wrong structure even for old-format data without requiring migration first.
Framework-Agnostic Deserialization JsonDeserializer<T> is a simple functional interface. Works with Jackson, Gson, or any JSON library. No compile-time dependency on any serialization framework.

8 Risks & Mitigations

RiskImpactMitigation
Developer forgets a migration step High — broken chain App fails to start immediately (fail-fast). Cannot reach production.
Migration has a bug (wrong field name) Medium — bad data in memory Schema validation catches it. Write unit tests for each migration.
Performance with many versions (e.g., 50 steps) Low — microseconds per step JsonPath operations are fast. Cache the DTO if read-heavy.
JsonPath library vulnerability Low — transitive dependency Library uses json-path 2.9+ (actively maintained). Bump via dependencyManagement.
Team doesn't define schemas Low — validation skipped Validation is opt-in. Migration still works. Add schemas incrementally.
